**The reorder engine is correct. This is a data problem, not a code problem.** The audit joined `supplier_products.csv` (purchase_unit, units_per_purchase_unit) with `products.csv` (inventory_unit) and `location_product_settings.csv` (target/max), and was independently reproduced against the real seed files: all 12 flagged rows match the source data exactly, zero hallucinated values, no false positives, and no collision row was missed.

12 of 134 supplier_product rows are flagged. The dominant defect: the sheet tracks each item in its base/recipe unit (kg, or szt-of-bread) while the supplier sells it in a multi-base PACK, but the seeder either (1) reused the SAME unit name for both `purchase_unit` and `inventory_unit` while keeping a ratio ≠ 1 — which the engine reads as "1 kg = 2 kg" or "1 szt = 15 szt" — or (2) stuffed a price-per-kg ratio into `units_per_purchase_unit`, producing impossible sub-1 values (P050 = 0.82, P052 = 0.72, P055 = 0.6).

**Hard rule:** if `purchase_unit == inventory_unit`, the ratio MUST be 1. Any ratio ≠ 1 on a same-named unit is a defect.

The location `target` value is the tiebreaker for the real base unit (P021 target 44 = 44 kg of fries; P024 target 10 = 10 kg of gyros; P026 target 5 = 5 opak/bags of pita). The seeder already did this correctly on 9 other rows by giving the pack a DISTINCT name (Tzatziki = wiadro/3, Feta = blok/2, Falafel = karton/5 → all into kg). Those rows are the templates the operator should copy.

### Two fix shapes
- **(a) Distinct pack name** — the item is genuinely tracked in a base unit but bought as a multi-base pack. Rename `purchase_unit` to a distinct name (blok / karton / opak) and keep `units_per_purchase_unit` as the true count of base units per pack. Applies to P024, P025, P026, P021, P022, P023, P057, P058.
- **(b) Ratio = 1** — the item is genuinely bought and counted in the same unit, and the ratio ≠ 1 was a price artifact. Set `units_per_purchase_unit = 1`. Applies to the impossible sub-1 spices P050, P052, P055.

Never leave `purchase_unit` identical to `inventory_unit` with a ratio other than 1, and never store a price ratio (< 1) in `units_per_purchase_unit`.

### Priority / verification flags (folded in from critique)
- **P026 — fix (a) ONLY.** The supplier note "1 opak = 12 szt" plus the name "Pita (opakowania) szt 10" prove a real 12× case→bag structure. Setting ratio = 1 here would destroy it and book one case as one bag. Rename to `karton`, keep ratio = 12. Ratio = 1 is NOT a legitimate option for pita.
- **P050 / P052 / P055 — deterministic, no operator judgment.** Single instruction: set ratio = 1 and delete the price-ratio note; do NOT rename the unit. The "ships as a fixed pack" alternative is a separate supplier-verification task, not a Sheet cell edit.
- **P023 — VERIFY WITH CAPTAIN.** Its supplier note is empty and target = 1.5 is also a plausible piece count; the kg conclusion is inference by analogy with P021/P022, not proof. The proposed fix is most likely right but should be confirmed, like other "verify with Captain" rows.
- **P025** is unused at WOLA (target = 0) — latent-only, lower priority than P024.
- **P038** is not a math break (ratio = 1 is correct); only its note implies a false 15× conversion and needs rewording.

### How to fix in the sheet (for the operator)
For each flagged row, work in `supplier_products.csv` and `products.csv`:
1. **Decide the real inventory unit** from the location target value and the supplier note (e.g. target 44 / "≈ 2.5 kg" → the item is kg).
2. **If the pack holds several base units** (Gyros blocks, frozen bags, pita cases, salt/pepper packs): give `purchase_unit` a DISTINCT name — `blok`, `karton`, or `opak` — and keep `units_per_purchase_unit` as the true count of base units per pack (15, 25, 12, 2.5, 2.27, 2, 4). Copy the format of the already-correct rows: Feta `blok`/2, Falafel `karton`/5, Tzatziki `wiadro`/3.
3. **If it is genuinely counted and bought in the same unit** (the sub-1 spices P050/P052/P055): set `units_per_purchase_unit = 1` and delete the price-ratio note. Do not rename the unit.
4. **Two hard don'ts:** never leave `purchase_unit` the same word as `inventory_unit` with a ratio other than 1; never put a price ratio (a value < 1) in `units_per_purchase_unit`.
5. **Before treating as done:** confirm P023 with the Captain (note is empty), and reword the P038 note so it does not imply a 15× conversion.
