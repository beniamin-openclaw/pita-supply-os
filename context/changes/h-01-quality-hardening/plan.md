# H-01 Quality Hardening (parallel-safe subset) Implementation Plan

## Overview

Deliver the **parallel-safe subset of roadmap H-01 (`quality-hardening`)**: a backend
dependency lockfile, CI hardening (the gate already runs the product — this makes it
reproducible + self-checking), and a thicker ruff ruleset. Strictly tooling/CI/config —
**no product code** — so it runs alongside the in-flight `demo-blocker-decimals-save`
(branch `claude/exciting-curie-e20933`) without collision. TS `strict` + mypy/pyright are
recorded as a **deferred phase** that runs only after the demo-blocker merges to `main`.

## Current State Analysis

- **No backend lockfile.** `supply-os-v1/pyproject.toml` `[project.dependencies]` and
  `supply-os-v1/requirements.txt` both use `>=` ranges. `requirements.txt` is a
  hand-maintained mirror; its own comment confirms **Railpack (Railway) installs from
  `requirements.txt`**, not from the PEP 621 pyproject deps. → builds are not reproducible.
- **Install sources already diverge.** CI's `backend` + `backend-integration` jobs run
  `pip install -e ".[dev]"` (pyproject, loose); Railway installs `requirements.txt` (loose).
  Nothing is pinned in either path.
- **CI already runs the product** (`.github/workflows/ci.yml`): `backend` (ruff + pytest),
  `backend-integration` (real Postgres 16 service → `pytest -m integration`, the Supabase
  path), `frontend` (`npm ci` → build → lint → vitest). The `lessons.md` "quality-gate.yml
  ran sibling tooling" gap is already closed — that file does not exist in this repo.
- **ruff config is bare**: `[tool.ruff]` has only `line-length = 100` + `target-version =
  "py310"`; no `[tool.ruff.lint] select`, so only ruff defaults (E/F) run.
- **Frontend is already reproducible**: `frontend/package-lock.json` + `npm ci`. Out of scope.
- **conftest** (`supply-os-v1/tests/conftest.py`) forces the seed backend and blanks all
  Google/Supabase creds before any `app.config` import (order-independence lesson). Untouched
  by this change.
- **Local toolchain**: `pip-tools` and `uv` are both **not installed**; Homebrew `python3` is
  **3.14**; CI pins Python **3.12**; ruff 0.15.16 + Homebrew node v25 available.

- A clearly-labeled **deferred phase** documents TS `strict` + mypy/pyright (+ `ruff format`,
  + any ruff families needing code fixes), gated on the demo-blocker merge.

## Implementation Approach

pip-tools compiles `pyproject.toml` into two pinned files: `requirements.txt` (runtime, from
`[project.dependencies]`) and `requirements-dev.txt` (runtime + dev, from `--extra dev`).
`pyproject.toml` stays the single loose source of truth; the two `.txt` files become generated
artifacts. CI installs the pinned dev lock then `pip install -e . --no-deps` for the local
`app` package, and a `lockfile-drift` job recompiles + diffs to keep the lock honest. ruff is
expanded empirically — enable a curated candidate set, then prune (whole families or single
rule codes via `ignore`) until `ruff check .` is clean with no code touched.

## Critical Implementation Details

- **One canonical Python 3.12 venv for lock-gen AND local verification (F2).** `pip-compile`
  resolves using the running interpreter's environment markers and stamps an autogen header with
  the exact command + Python version; CI recompiles on `setup-python` 3.12. A 3.14-generated lock
  (Homebrew default here), a different pip-tools version, or a different command string would all
  make the `lockfile-drift` job's `git diff` fail on a no-op. So: create ONE Python 3.12 venv and
  use it for both lock generation and the backend `ruff`/`pytest` verification — local then matches
  CI byte-for-byte. Obtain 3.12 via `python3.12`, `pyenv`, a venv, or `brew install python@3.12`
  (last resort); if none is reachable, generate the lock inside a one-off CI run. Homebrew's 3.14
  is an acceptable *fallback* only for running the version-tolerant test suite — never for
  generating the committed lock. This is the change's main execution risk.
- **Document the exact `pip-compile` invocation CI mirrors.** The committed locks and the
  `lockfile-drift` job must use the identical command (same flags, e.g. any `--strip-extras`) and
  the pinned pip-tools — record both commands verbatim in root AGENTS.md so a future regen can't
  drift the header/ordering.
- **Pin the resolver for a deterministic drift guard.** Add `pip-tools` to the dev extra so its
  version is itself pinned into `requirements-dev.txt`; the `lockfile-drift` job installs that
  lock (getting the same pip-tools) before recompiling, so local and CI resolutions match.
- **Lock generation needs PyPI access.** If the implement sandbox blocks network, generate in a
  networked context or have the owner run the documented command; the rest of the change is
  offline-safe.

## Phase 2: CI wiring + lock-drift guard

### Overview

Make the lockfile load-bearing: CI installs from it, and a drift job prevents it silently
diverging from `pyproject.toml`. Prove the whole gate green.

### Changes Required:

#### 1. Backend jobs install from the lock

**File**: `.github/workflows/ci.yml`

**Intent**: Switch the `backend` and `backend-integration` jobs from `pip install -e ".[dev]"`
to a reproducible install from the pinned dev lock plus the local package.

**Contract**: Replace each "Install dependencies" step with
`pip install -r requirements-dev.txt` then `pip install -e . --no-deps`. Set
`cache-dependency-path: supply-os-v1/requirements-dev.txt` on each job's `setup-python` so the
pip cache key tracks the lock. Test/lint steps unchanged; the integration job's env
(`SUPPLY_OS_DATA_BACKEND=supabase` + DSN) unchanged.

#### 2. Lock-drift guard job

**File**: `.github/workflows/ci.yml`

**Intent**: Add a job that recompiles both locks and fails if the committed files are stale,
so a `pyproject.toml` dependency edit without a regen is caught.

**Contract**: New job `lockfile-drift` (working-directory `supply-os-v1`, Python 3.12): install
`requirements-dev.txt` (for the pinned pip-tools), run both `pip-compile` commands from Phase 1,
then `git diff --exit-code requirements.txt requirements-dev.txt`. Non-zero diff → fail.

### Success Criteria:

#### Automated Verification:

- Backend gate green locally, in the canonical Python 3.12 venv: `cd supply-os-v1 && python -m ruff check . && python -m pytest -q`.
- Frontend gate green locally: `cd frontend && PATH=/opt/homebrew/bin:$PATH npm ci && npm run build && npm run lint && npm run test`.
- Drift guard is a no-op on a fresh checkout: recompiling both locks produces no `git diff`.
- `ci.yml` is valid YAML and the backend jobs reference `requirements-dev.txt` (not `-e ".[dev]"`).

#### Manual Verification:

- Reading `ci.yml` confirms the gate runs `supply-os-v1` ruff+pytest (+ integration) and `frontend` build+lint+vitest — the DoD "PROVEN it runs the product."
- **Real CI proof (the DoD evidence):** a push of the *feature branch* `claude/jolly-mestorf-a20182` triggers the workflow (`on: push:` fires for any branch) and all jobs go green on the Actions runner — exercising the real pip cache keys + the Postgres service job that local runs can't. This is NOT a push to `main` (the only gated target), but it IS a remote action, so it is **permission-gated**: the agent pushes the branch only when the owner asks. Local runs are the pre-push gate; the branch-push Actions run is the actual evidence (capture the run URL).
- The drift job would fail on a deliberate `pyproject.toml` edit-without-regen (reasoned, not necessarily executed).

**Implementation Note**: Pause after the local green-run proof. The true DoD evidence is a
feature-branch push that turns the Actions run green; request the owner's go-ahead to push the
branch (never `main`), then capture the run URL.

## Phase 3: Thicker ruff ruleset (clean-only)

### Overview

Raise lint coverage for `supply-os-v1` to a curated modern set, keeping everything that passes
clean today with zero product-code changes.

### Changes Required:

#### 1. Expanded ruff lint config

**File**: `supply-os-v1/pyproject.toml`

**Intent**: Add `[tool.ruff.lint]` with a curated `select` and targeted `ignore`s so the ruleset
catches more issues going forward while `ruff check .` stays 100% clean now.

**Contract**: Empirical method — for each candidate family run
`/opt/homebrew/bin/python3 -m ruff check --select <FAM> .` and keep it only if it passes clean,
or enable it with `ignore` for the specific flagging codes (config-only grandfathering).
Candidate families: `UP, B, C4, SIM, PIE, RET, RUF, ISC, PT, TID, G, W` (plus existing `E, F`).
Expected `ignore`: at least `B008` (FastAPI `Depends`). Families whose only fix is product-code
edits (`I` isort reorder, `PTH` pathlib) are **not** enabled here → Phase 4. The final
`select`/`ignore` is whatever makes `ruff check .` exit 0 with no `.py` file modified.

### Success Criteria:

#### Automated Verification:

- `cd supply-os-v1 && /opt/homebrew/bin/python3 -m ruff check .` exits 0.
- `git status --porcelain 'supply-os-v1/app/**/*.py' 'supply-os-v1/tests/**/*.py'` shows no modified product/test `.py` files (config-only).
- The CI backend ruff step (now exercised against the new config) passes locally.

#### Manual Verification:

- The enabled `select` is a meaningful uplift over E/F (not just isort), and every `ignore` has a one-line reason comment.
- No `# noqa` was added to any product file (would be a code edit).

**Implementation Note**: This is the last executing phase; after it, re-run the full

## Testing Strategy

### Unit / suite:

- Backend: `cd supply-os-v1 && /opt/homebrew/bin/python3 -m pytest -q` (default suite, seed backend via conftest) stays green against the locked deps.
- The real-Postgres integration suite is exercised by CI's `backend-integration` job (not run locally; no local Postgres assumed).

### Integration / gate:

- Full local gate dry-run mirroring CI: backend ruff + pytest; frontend `npm ci && npm run build && npm run lint && npm run test` (Homebrew node).
- Lock-drift no-op: recompile both locks on Python 3.12 → empty `git diff`.

### Manual Testing Steps:

1. Fresh-venv install from `requirements-dev.txt` + `pip install -e . --no-deps` → import `app.main` succeeds.
2. Read `ci.yml`: backend jobs install from the lock; `lockfile-drift` job present.
3. Spot-check pinned versions vs current resolution for the runtime-critical packages.
4. Confirm no product `.py`/`.ts` file is modified by any phase (`git status`).

## Performance Considerations

Negligible. CI gains one short `lockfile-drift` job (a single `pip-compile` pass) and the backend
jobs install from a pinned lock (typically faster + better cached than range resolution).

## Migration Notes

- **Railway blast radius:** the next deploy after merge installs the **pinned** `requirements.txt`.
  Versions equal what currently resolves, so risk is low; the owner gates the merge. Rollback =
  revert the two `.txt` files (or unpin) and redeploy.
- **Regen procedure** (documented in root AGENTS.md): on Python 3.12, run the two `pip-compile`
  commands and commit the diffs; the `lockfile-drift` CI job enforces this.
- **Hashes:** `--generate-hashes` deferred — can be added later for stricter integrity once the
  plain-pin path is proven on a Railway deploy.
