# H-01 Quality Hardening (parallel-safe subset) — Plan Brief

> Full plan: `context/changes/h-01-quality-hardening/plan.md`

## What & Why

Deliver the parallel-safe slice of roadmap **H-01**: pin the backend's dependencies, make CI
install from that lock and self-check it, and thicken the ruff ruleset. The repo has no backend
lockfile (`pyproject` + `requirements.txt` both use `>=`), so neither CI nor Railway builds are
reproducible — `health-check.md` Fix #3/#5. Strictly tooling/CI/config so it can run beside the
in-flight `demo-blocker-decimals-save` without touching its files.

## Starting Point

`ci.yml` **already** runs the product (backend ruff+pytest, a real-Postgres integration job,
frontend build+lint+vitest) — the old `quality-gate.yml` gap is already closed. But the backend
installs deps via loose `pip install -e ".[dev]"` (CI) and a loose hand-mirrored
`requirements.txt` (Railway/Railpack), and ruff runs only its E/F defaults. Frontend is already
reproducible (`package-lock.json` + `npm ci`).

## Desired End State

`requirements.txt` (Railway runtime) + a new `requirements-dev.txt` (CI) are fully pinned,
generated by `pip-compile` from `pyproject.toml` (kept as the loose source of truth). CI backend
jobs install from the lock; a `lockfile-drift` job fails on a stale lock. `ruff check .` passes
100% clean under a curated, expanded ruleset with **zero** product-code edits. TS `strict` +
mypy are documented as a deferred phase, gated on the demo-blocker merge.

## Key Decisions Made

| Decision | Choice | Why (1 sentence) | Source |
| --- | --- | --- | --- |
| Lockfile tool | pip-tools, compiled from pyproject | Reproducible deploy + CI, idiomatic pip, no new runtime tool; Railpack reads requirements.txt | Plan |
| Blast radius | Pin requirements.txt (Railway) + requirements-dev.txt (CI) | Closes #3 fully (repro in both deploy and CI); owner gates the merge | Plan |
| CI depth | Wire to lock + drift guard | Makes the lock load-bearing and self-checking | Plan |
| Ruff scope | Curated families, clean-only + targeted ignores; defer formatter | Real lint uplift with no product-code churn (parallel-safe) | Plan |
| TS strict / mypy | Deferred to a post-merge phase | Would light up files the demo-blocker is editing | Plan |
| Lock Python | Generate on 3.12 (match CI), pin pip-tools | Deterministic drift guard | Plan |

## Scope

**In scope:** backend lockfile (pip-tools); CI install-from-lock + drift job; expanded ruff
config; doc update of the "no lockfile" gap; deferred-phase record.

**Out of scope:** any product `.py`/`.ts` (incl. all demo-blocker files); TS strict; mypy/pyright;
`ruff format` enforcement; ruff rules needing code fixes (isort/pathlib); frontend lockfile;
`--generate-hashes`; secrets; pushing to `main`.

## Architecture / Approach

`pyproject.toml` (loose, source of truth) → `pip-compile` → pinned `requirements.txt` (runtime)
+ `requirements-dev.txt` (runtime+dev). Railway installs the runtime lock; CI installs the dev
lock then `pip install -e . --no-deps`. A `lockfile-drift` CI job recompiles and `git
diff --exit-code`. ruff expanded empirically until clean.

## Phases at a Glance

| Phase | What it delivers | Key risk |
| --- | --- | --- |
| 1. Backend lockfile | Pinned runtime + dev locks from pyproject | Must generate on Python 3.12 (local is 3.14); needs PyPI access |
| 2. CI wiring + drift guard | Backend installs from lock; drift job; green proof | Drift determinism (pinned pip-tools + 3.12) |
| 3. Thicker ruff | Expanded clean-only ruleset | Keeping it config-only (grandfather via ignore, never noqa) |
| 4. DEFERRED: TS strict + mypy | Documented, not executed | Gated on demo-blocker merge |

**Prerequisites:** Python 3.12 interpreter + PyPI access for lock generation; Homebrew node for FE.
**Estimated effort:** ~1 session across phases 1–3; phase 4 is a separate future change.

## Open Risks & Assumptions

- **Lock determinism across Python versions** — local 3.14 vs CI 3.12 could drift; mitigated by generating on 3.12 + pinning pip-tools. Main execution risk.
- **PyPI access in the implement sandbox** — if blocked, generate in a networked context or have the owner run the documented command.
- **Railway redeploy uses the new pins** on merge — versions match current resolution, owner gates the merge, rollback = revert the `.txt` files.

## Success Criteria (Summary)

- Backend deps are pinned + drift-guarded; a fresh-venv install from the lock runs the suite green.
- CI gate is demonstrably green and runs `supply-os-v1` (ruff+pytest) + `frontend` (build+lint+vitest).
- `ruff check .` passes under a thicker ruleset with no product-code changes; TS strict + mypy are queued as a clearly-gated deferred phase.
